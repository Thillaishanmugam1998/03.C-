---

### Rule 2 — Comments vs Console Output (VERY IMPORTANT)

COMMENTS (use // inside regions):
- Use multi-line // comments to explain concepts, theory, and gotchas
- This is where all teaching content lives
- Do NOT print theory or explanations inside Console.WriteLine

CONSOLE OUTPUT:
- Use Console.WriteLine() ONLY to print values, results, labels, and separators
- Console output should show WHAT the code produces, not explain WHY

WRONG example:
Console.WriteLine("int is a value type and stored in stack memory");  ← NO

CORRECT example:
// int is a value type — stored on the Stack memory
// Range: -2,147,483,648 to 2,147,483,647
Console.WriteLine($"int Max Value: {int.MaxValue}");   // Output: int Max Value: 2147483647

---

### Rule 3 — Expected Output Comment on Every Console.WriteLine

Every single Console.WriteLine must have its expected output written as an inline comment:

Console.WriteLine($"double value: {d}");        // Output: double value: 3.14159265358979
Console.WriteLine($"decimal value: {m}");       // Output: decimal value: 3.14159265358979323846

## 📌 DETAILED CONTENT FOR EACH REGION

### #region 01 — int, long
- Declare and initialize int and long variables
- Show int.MaxValue, int.MinValue, long.MaxValue
- Show overflow example (what happens when you exceed int range)
- Show when to use int vs long
- Common mistake: using int for database IDs that can grow very large
- Real-world analogy: int = city population counter, long = world population counter over centuries

### #region 02 — float, double
- Declare float and double variables
- Show precision difference with same decimal number
- Show float suffix 'f' requirement
- Show double.MaxValue
- Common mistake: comparing floats with == (floating point imprecision)
- Real-world analogy: float = good enough for game character position, double = scientific calculations like GPS coordinates

### #region 03 — decimal
- Declare decimal variable with 'm' suffix
- Show precision vs double with same value
- Show decimal.MaxValue
- Demonstrate why decimal is used for money (no rounding errors)
- Common mistake: using double for financial calculations
- Real-world analogy: bank transaction amounts, tax calculations, invoice totals

### #region 04 — bool
- Declare bool variables (true / false)
- Show bool in a simple if condition
- Show bool result from comparison operators
- Common mistake: writing if(isActive == true) instead of if(isActive)
- Real-world analogy: light switch (on/off), user login status (logged in / not logged in)

### #region 05 — char
- Declare char with single quotes
- Show char numeric value (casting to int)
- Show char comparison
- Common mistake: using double quotes for char (that's a string!)
- Real-world analogy: a single letter on a keyboard key, grade result ('A', 'B', 'C')

### #region 06 — string
- Declare string, show it is a reference type
- Show string immutability (new string created on modification)
- Show string.IsNullOrEmpty, string.Length, string.ToUpper
- Show string interpolation and concatenation
- Show null string behavior
- Common mistake: using + for string concatenation in loops (use StringBuilder instead)
- Real-world analogy: a printed document — every edit creates a new copy, not modifies the original

### #region 07 — var
- Show var with different types (int, string, double, list)
- Show that var is still strongly typed (compiler infers type)
- Show var limitation: cannot declare without initialization
- Common mistake: thinking var is like JavaScript's dynamic typing — it is NOT
- Real-world analogy: ordering food without saying the name — waiter infers from your description, but it's still one specific item

### #region 08 — object
- Show object holding int, string, bool values (boxing)
- Show casting back from object to original type (unboxing)
- Show object.GetType() to check actual type at runtime
- Show performance concern with boxing/unboxing
- Common mistake: overusing object type instead of generics
- Real-world analogy: a universal storage box — you can put anything in, but you must label it to get the right thing back

### #region 09 — dynamic
- Declare dynamic variable
- Show dynamic changing type at runtime
- Show dynamic method call (resolved at runtime)
- Show difference between var (compile-time) and dynamic (runtime)
- Show that dynamic can cause runtime errors that var cannot
- Common mistake: using dynamic everywhere "for flexibility" — it removes type safety
- Real-world analogy: var = GPS with fixed route at start, dynamic = driving without GPS and deciding at each turn

### #region 10 — Value Types vs Reference Types
- Show value type copy behavior (int) — changing copy does NOT affect original
- Show reference type copy behavior (class/string) — both point to same object
- Show side-by-side comparison with clear Console output
- Explain struct = value type, class = reference type
- Show null assignment — only reference types can be null (before nullable value types)
- Common mistake: expecting value type behavior from reference types
- Real-world analogy:
  Value type = photocopy of a document (you edit your copy, original unchanged)
  Reference type = shared Google Doc (both people see the same changes live)

### #region 11 — Stack vs Heap
- Explain in comments what Stack is (automatic, fast, LIFO, limited size)
- Explain in comments what Heap is (manual/GC managed, slower, large, flexible)
- Show which types go where:
  Stack: int, double, bool, char, struct, local variables
  Heap: string, object, class instances, arrays
- Show a practical code example where a local int goes to Stack and a class instance goes to Heap
- Show how reference variable itself is on Stack but the object it points to is on Heap
- Add an ASCII diagram in comments showing Stack and Heap side by side:

//  Stack                    Heap
//  ──────────────           ──────────────────────
//  | age = 25    |          | Person { Name="Raj" }|
//  | name = ref ─────────► |                      |
//  | isActive=true|         |                      |
//  ──────────────           ──────────────────────

- Common mistake: thinking string is a value type because it "feels" like one
- Real-world analogy:
  Stack = your office desk (fast to grab, limited space, auto-cleaned when you leave)
  Heap = company storage room (large, organized by GC cleaner, accessed by reference number)
